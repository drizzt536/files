Strings will be length-prefixed strings (LP strings) (a variation on Pascal strings), with a 64-bit length field instead of an 8-bit field. The pointer should point to the first character, and the length will be at `qword [ptr - 8]`. Most (all?) stdlib functions will expect this type as arguments, and will return this type, except in select scenarios. Essentially, `typedef struct {u64 n; char p[]; } string;` and then point to `p` instead of the actual start of the struct.

This construction has a main downside, which is inline tokenization, so for tokenization, users should instead use a string view, or `struct { u64 n; char *p; }`. To convert from an LP string to a string view, just grab the length from the string prefix, e.g. `mov r8, qword [rax - 8]`. To temporarily convert from a string view to an LP string, swap the length register with the data before the start of the string, e.g. `xchg r8, qword [rax - 8]`, and repeat that instruction to convert back. This assumes that whenever a string view is used, it was originally an LP string. When creating a string view from scratch, if it ever needs to be converted to an LP string, then the first character should be at least 8 bytes after the base of the memory allocation to avoid potential memory corruption.

One of the main rationales for this design (LP strings as the default) is that string views take up 128 bits, and LP strings take up 64-bits, so functions that take string views are more likely to spill onto the stack, which goes against the design choices of the ABI.
