strange values:
NOTE: 2A and AA are ignored, and 37 and B7 map to the actual print screen codes.
E0 2A E0 37: print screen pressed  => 0x54
E0 B7 E0 AA: print screen released => 0xD4
E1 1D 45: pause break pressed      => 0x55
E1 9D C5: pause break released     => 0xD5

basically, there is no point to implement the whole state machine because that whole aspect
of the standard is retarded. and since pause break is the only thing that uses E1 as the first
byte, if it sees 0xE1, then it will just consume the next bytes, and add either 6D or ED to the key ring, depending on if 45 or C5 is the last byte.

the following other things map to zero since they don't exist:
	00, 55, 56, 59-80, D5, D6, D9-FF

the flattening of the single-byte scancodes is just so there aren't multiple scancodes that
mean almost identical things. Like nobody cares if you use left shift vs right shift. they are both shift.

PrtSc maps to the same thing as SysRq because my keyboard sends alt+SysRq when I do alt+PrtScr,
so I assumed that they are the same thing.

conversion from the flattened scancode to ASCII will be in the kernel and not in ISR 33.
